# 文法道場 学習設計（アーキテクチャ）

> ドリル1問の作り方は `grammar-drill-design.md`（v2基準）。本書はその上位＝**学習効果の設計**。

## 前提

ゴールは「文法クイズに正答できる」ではなく **「本物の古文を正確に読める」**。
文法が入試で効く場面は3つに尽きる：

1. **現代語訳の正誤** — 「ぬ」「に」「なり」の読み違いは訳を180°変える
2. **識別問題** そのもの
3. **敬語 → 主語復元**（誰が誰に、の判定）

学習効果の3条件と現状：

| 条件 | 学習科学 | 現状 |
|---|---|---|
| 想起 | 検索練習＋間隔反復 | ✅ SRS（srs_state流用）あり |
| **弁別** | **交互練習（interleaving）** | ❌ **単元ブロック練習は単元名が答えを漏らす**。「完了ぬ」の単元内で「このぬは？」→ほぼ完了に決まっている |
| 転移 | 練習＝本番の処理（transfer-appropriate） | ❌ 作問例文のみ。本物の文で使う場がない |

## 設計：三層「型 → 混ぜる → 実戦」

### L1 型を作る — 単元ドリル（v2・既存路線）
単元ごと8問・4段階（かたち→つなぐ→よみとく→まぎらわし）。
役割は**判別手順（スキーマ）の獲得と成功体験**。ここで完成度を追わない——弁別はL2の仕事なので、L1は8問で切り上げて先へ進ませる。

### L2 混ぜて鍛える — 識別クラスター（本丸）
章立てを「助動詞ごと」でなく**「紛らわしさごと」**にした横断ドリル。単元名のヒントが消え、初めて本物の弁別が起きる。

| クラスター(topic_id) | 対決させる正体 |
|---|---|
| shikibetsu-ni | 格助詞に／接続助詞に／断定なり連用形／完了ぬ連用形／ナリ活用語尾／副詞の一部 |
| shikibetsu-nu-ne | 完了ぬ⇔打消ぬ、完了ね⇔打消ね、ナ変語尾 |
| shikibetsu-namu | 強意ぬ＋む／願望の終助詞／係助詞／ナ変未然＋む |
| shikibetsu-ru-re | 受身尊敬る・らる／完了り／四段已然形語尾 |
| shikibetsu-nari | 断定／伝聞推定／動詞成る／ナリ活用語尾 |
| shikibetsu-shi | 過去き連体形／サ変連用形／副助詞し／形容詞語尾 |

- 既存リファレンス `shikibetsu.json`（紛らわしい21語）が解説の受け皿として既にある
- **スキーマ変更不要**：新しい topic_id を増やすだけで道場ホームに自動で並ぶ
- 解禁条件（将来）：構成単元のL1を済ませたら出現、でもよい

### L3 実戦で使う — 本文識別（このアプリ固有の強み）
`public/texts` の本文は**全トークンに活用形タグが付与済み**（誤タグ修正の蓄積で品質も高い）。ここから識別問題を機械生成する：

- 本文から「ぬ」「に」「なむ」「る」「なり」等の出現箇所を抽出 → 「この文のこの語の正体は？」を自動作問（正解＝既存タグ）
- 生徒が**授業で読む本文そのもの**なので文脈知識が転移を助ける
- 解答後は TextReader の該当箇所へリンク（読解層と道場が接続する）
- 作問コストゼロでスケールする。スクリプトで生成→人間が検品→JSON投入

### 横串

- **SRS復習が自然な interleaving 装置**：`/read/grammar/review` は既に全単元シャッフル。L2/L3 の qid も同じ箱に入るので、復習は自動的に「型と識別と実戦の混合」になる
- **弱点モード**（将来）：word_stats の誤答集計から最も落とす識別だけを出す。教師側でも「学年がどこで落ちるか」が見える＝教材への一次入力
- **記述式（将来・小改修）**：「書く」を已然形に→入力させて完全一致採点。選択肢の再認より生成効果が強い。DrillSession に input 型を1つ足すだけ
- **敬語は別設計**：暗記でなく「敬意の方向（誰から誰へ）」「主語復元」を問う。L3（本文に敬語タグ）と最も相性が良い

## 効果の根拠（1行ずつ）

- 検索練習・間隔反復 → SRS（実装済）
- **交互練習** → L2クラスター＋横断復習（ブロック練習だけでは弁別は育たない）
- **転移適合** → L3本文識別（練習の処理＝本番の処理）
- 生成効果 → 記述式（将来）
- 誤りからの学習 → 誤答時のみ解説表示＋判別手順1行（v2基準）
