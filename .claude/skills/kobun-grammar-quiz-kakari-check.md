# kobun-tan 文法道場クイズ 係り結び手がかり点検スキル

## 背景

`supabase/seeds/grammar-corpus-lv23.json` 等の **コーパス機械生成ドリル** は、活用形クイズ
（`kind: "katsuyo-fill"` など）で本文に次の2種のマークを埋め込む。

- `【…】` … 設問の対象語（活用形を問う語）
- `《…》` … 手がかり（下線が引かれる語）

生成器は手がかりを **常に「対象語の直後の語」** に置き、説明も
「**下に続く下線の語から判断して連体形／已然形**」で一律に付ける。

ところが活用形が **係り結び** で決まる場合、活用形を決めるのは **前にある係助詞** であって
後続語ではない。よって手がかりも説明も誤りになる。

| 係助詞（対象語より前） | 結びの活用形 |
|---|---|
| ぞ・なむ（なん）・や・か | 連体形 |
| こそ | 已然形 |

（疑問・反語の副詞「いかが」「いかで」「など」なども連体形結びに準ずる）

### 実例 (2026-06-14 検出・修復)

`grammar-corpus-lv23.json` の6問。いずれも **答え（活用形）は正しい** が手がかり・説明が誤り。

| id | 本文 | 答 | 誤手がかり→正 |
|---|---|---|---|
| jodoshi-keri-x05 | かう**こそ**燃え**けれ** | 已然形 | 《と》→《こそ》 |
| jodoshi-mu-x07 | 飽かず**や**あら**む** | 連体形 | 《、また》→《や》 |
| jodoshi-mu-x08 | **いかが**はせ**む** | 連体形 | 《と》→《いかが》 |
| jodoshi-mu-x10 | 命**こそ**絶え**め** | 已然形 | 《さだめなき》→《こそ》 |
| jodoshi-mu-x11 | 心に**こそ**あら**め** | 已然形 | 《否び》→《こそ》 |
| jodoshi-mu-x12 | 目に**こそ**見えたまは**め** | 已然形 | 《わ》→《こそ》 |

## 検出方法

```
python scripts/check-grammar-quiz.py            # 既定6シードを走査
python scripts/check-grammar-quiz.py path.json  # ファイル指定
```

結果は `scripts/grammar-quiz-report.json` に出力。標準出力に件数と要約。

### 判定ロジック（スクリプト内）

- **已然形**：対象語より前に「こそ」があり、手がかり《》が「こそ」でなく、
  後続語が「ば／ど／ども」（已然形＋接続助詞）でもない → 係り結び誤り。これは高精度。
- **連体形**：対象語より前に ぞ／なむ／や／か があり、かつ後続語《》が連体形を
  単独で正当化しない（句読点始まり・引用「と」・「また」・空）場合のみ誤りと判定。
  - **重要**：連体形は「後続の体言」「断定『なり』」「接続助詞『に』」でも正当に説明できる。
    その場合『下に続く語』の説明は誤りでないので **フラグしない**（誤検知抑制）。

### 誤検知に注意（手動精査が必須）

スクリプトは表層文字列マッチなので、精査して切り分けること。よくある **正常パターン**：

- `かり`（形容詞連用）`いか`（いかが）`書か`（未然形）`かは`(虫) 等に含まれる「か」は係助詞でない。
- `〜めやは`／`〜めやも`：已然形＋反語の係助詞「や」は **後続語が正しい手がかり**（こそ不要）。
- 連体形＋体言（連体修飾）／＋断定「なり」／＋接続助詞「に」は後続語が正しい手がかり。

## 修正手順

1. 該当 item の `context` で `《…》` を **係助詞（こそ／や／ぞ／なむ／か／いかが）** に移す。
   - 対象語の直後にあった旧マークは外す（本文の文字列自体は変えない）。
2. `explanation` の「下に続く下線の語から判断して◯◯形。」を
   - 已然形：「係助詞「こそ」の係り結びで已然形。」
   - 連体形：「係助詞「や／ぞ／なむ／か」の係り結びで連体形。」
   - 疑問副詞：「疑問の副詞「いかが」に呼応して連体形（係り結びに準ずる）。」
   に置換（訳・出典部分は残す）。
3. `python -c "import json;json.load(open(...))"` で JSON 妥当性確認 → チェッカー再実行で flagged=0 を確認。

## DB 反映（本番）

seed 修正はローカルのみ。本番（Supabase）に反映するには id で upsert：

```
node --env-file=.env.local scripts/apply-drills.mjs supabase/seeds/grammar-corpus-lv23.json --merge
```

`--merge` は topic を削除せず id で upsert するため、変更行だけ安全に更新できる。
（`--merge` なしは topic 全削除→再投入。本番反映は外向き操作なので実行前に必ず確認すること。）

## 総合監査（全ドリル横断）

`scripts/audit-grammar-all.py` で全 seed を横断検査：

```
python scripts/audit-grammar-all.py   # scripts/audit-report.txt に出力
```

検出項目と判断の目安：
- **DUP_ID**（重複id）＝実バグ。apply-drills は id 単位 upsert なので衝突＝片方消失。
  例: nu/zu の v1(grammar-jodoshi-batch1.json) と v2(grammar-nu-zu-v2.json) の id 重複。
  → v1 を削除して解消（2026-06-15 実施済み）。
- ANSWER_NOT_IN_CHOICES / TOO_FEW_CHOICES / DUP_CHOICE / MARKUP_UNBALANCED ＝実バグ。
- **KATSUYO_ANS_ODD**＝多くは誤検知。`katsuyo-fill` には「活用形名を答える(連用形…)」型と
  「活用形を作って答える(書か…)」型があり、後者は答が語形なので正常。
- **ANS_NOT_IN_EXPL**＝多くは誤検知（説明が訳語を使い意味ラベル語を含まないだけ）。
- **KANYU_KAKUJUTSU_REVIEW**＝ぬ・つで推量語/命令形が対象語に連接し「完了」のもの。
  文末・命令の確述は「強意」が妥当（[[kobun-nu-tsu-kanyu-kakujutsu]] 参照）。文中の
  連用形完了（に・て＋後続語）は完了で正しい。
